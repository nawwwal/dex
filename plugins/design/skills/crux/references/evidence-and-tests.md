# Evidence and Tests

Use this when the claim needs validation design, user research, success criteria, or evidence audit.

## Assumption Graph

Track these internally:

- must be true
- nice to be true
- unknowns
- contradictions
- riskiest assumption
- evidence needed
- confidence
- what would change my mind

Do not dump the full graph unless the user asks for a deeper interrogation. Use it to choose the weak joint and the crux question.

## Pramana Ledger

Use one consistent evidence taxonomy:

- perception: directly observed behavior or artifact
- inference: conclusion drawn from signals
- analogy: comparison to another product, behavior, or market
- testimony: what someone said
- taste: preference or aesthetic judgment
- social proof: borrowed confidence from other people's behavior or status
- memory: recalled prior context
- assumption: currently unsupported claim

Most bad product thinking comes from treating inference like observation.

## Crude Test Design

Prefer tests that are informal, fast, and falsifiable:

- prototype
- fake-door test
- interview prompt
- concierge workflow
- landing page
- manual service
- paper prototype
- one-screen model
- spreadsheet simulation
- five-user observation

Specify:

- crude v1 or test method
- success signal
- failure signal
- desirability signal: does anyone care enough to change behavior?
- feasibility signal: can we deliver the crude version with current means?
- viability signal: does the value justify the cost, effort, risk, or business tradeoff?
- legibility signal: can the user understand what it is, what it can do, and what it expects? Mandatory for product or design claims.
- time to test
- what it teaches

Avoid:

- "Would you use this?" future-intention questions
- vague advice like "interview users"
- tests that prove polish instead of need
- tests that make the idea look mature before proving it matters
