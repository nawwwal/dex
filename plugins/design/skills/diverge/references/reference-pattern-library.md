# Reference Pattern Library

Use this as a mental library structure. References are optional and must inform a design layer.

Each reference should be stored or invoked as:

- Name:
- Domain:
- What problem it solves:
- Mechanism:
- Transferable principle:
- UI layer:
- Product layer:
- Example product use:
- Failure mode:

## Categories

- Product UX patterns
- Dashboard patterns
- Data visualization patterns
- Onboarding patterns
- Empty/error/loading patterns
- Copywriting patterns
- Typography systems
- Color systems
- Motion systems
- Interaction patterns
- Games
- Architecture
- Editorial design
- Film
- Music
- Art
- Ritual
- Tools
- Industrial/product design

## Rule

Study artists and non-design references for mechanics, not aesthetics. Artists can help question assumptions. Designers often provide known answers. Use both.

## Example entries

Name:
GitHub pull request diff

Domain:
Developer tooling

What problem it solves:
Reviewing before/after changes with accountable approval.

Mechanism:
Diff, comments, checks, ownership, approval, merge.

Transferable principle:
Trust comes from making change inspectable before commitment.

UI layer:
Trust model, diff view, action gating.

Product layer:
Permission review or schema change approval.

Example product use:
Connector scope changes show old permission, new permission, affected agents, test result, approve/reject.

Failure mode:
Looks too technical for non-technical users.
