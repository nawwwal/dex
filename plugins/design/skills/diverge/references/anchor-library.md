# Optional Reference Handling

References are optional. They are not required in every output.

The skill may use:

- Product references
- UI references
- Interaction references
- Copy references
- Typography references
- Color references
- Motion references
- Art references
- Non-software references

But every reference must pass this gate:

```md
Reference:
Observed mechanism:
Design layer it informs:
Product translation:
Concrete execution:
What not to copy:
```

## Example

Reference:
GitHub pull request diff

Observed mechanism:
Before/after comparison with reviewable change.

Design layer it informs:
Trust model, permission review, schema changes.

Product translation:
Connector permission changes become reviewable diffs.

Concrete execution:
Show old scope, new scope, affected agents, approve/reject, test connection.

What not to copy:
Do not make the UI look like code review unless the users are technical.

## Bad vs good

Bad:
"Make it like a garden."

Good:
"Use a growth metaphor only if the product has cumulative progress, seasonal change, care actions, and visible consequences over time."

## Legacy output to remove

- Domain quota rules
- Decorative reward references
- Poetic direction names
- Reference cards as mandatory output
- Reward cards as mandatory output

## Keep

- References as optional mental library
- Cross-domain study as a creativity tool
- Artist references only when they change composition, pacing, contrast, materiality, or interaction

## Rule

Study references for mechanics, not aesthetics. A reference that does not change execution should be deleted.
