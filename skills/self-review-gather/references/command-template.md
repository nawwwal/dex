## Execution Steps

Follow these steps in strict order. Decisions are autonomous — do NOT ask questions during collection.

---

### Step 1: Dedup Index

Read `~/.claude/career/case.md`. Build a dedup set:
- For each entry with an embedded URL: `sha256(normalize(artifact_url))` → primary key
- For each entry without URL: `sha256(normalize(workstream) + normalize(date_bucket) + normalize(first_8_words))` → secondary key

**Normalization**: lowercase, strip trailing slashes, collapse whitespace, remove query params from URLs. Apply consistently before hashing so `figma.com/file/abc?t=123` and `figma.com/file/abc` produce the same key.

Store both sets in memory for all subsequent steps.

---

### Step 2: Vault Direct Reads (always — minimum viable path)

Read the following files and extract evidence candidates:

**`~/.claude/memory/goals.md`** — Evidence Log sections (two formats):
- `### Week of YYYY-MM-DD — title` (manual weekly entries — highest quality; contain `[solid]`/`[evolving]` confidence tags)
- `### YYYY-MM-DD run` (auto-generated by `/think emerge` — useful for cross-referencing)
- Extract entries labeled by competency area or tagged `[solid]` / `[evolving]`
- Treat `[solid]` as high-priority candidates; `[evolving]` as medium-priority

**`~/.claude/memory/nawal-through-others.md`** — third-person signals:
- Read for external observations about the user's work (peer feedback, manager comments, stakeholder reactions)
- Do NOT use as standalone evidence — use to add corroborating framing to vault/DevRev candidates that cover the same work
- Especially useful for Leadership and Communication competencies

**`~/.claude/memory/decisions.md`** — Decision log:
- Find entries with `^dec-` block IDs within the review period (format: `^dec-YYYY-MM-DD-slug` at end of heading)
- Each decision entry = potential ownership evidence (selected design direction, scope change)
- Map to competency: design decisions → UX/Product Thinking; tool/process decisions → Leadership; cross-team decisions → Communication

**`~/.claude/log/` session journals** — scan files matching `YYYY-MM-DD-*.md` within the review period:
- Look for sections: `## Evidence Log`, `### Week of YYYY-MM-DD`, competency-tagged bullets
- Extract entries with external artifact links (Figma, DevRev, GitHub, Slack) — these are strongest
- Entries with `[solid]` confidence > `[evolving]` > untagged

**Apply dedup**: for each extracted item, check against the dedup index (primary key first, secondary key if no URL). Skip if already in `career/case.md`.

**Apply redaction** (all sources): replace customer/merchant names with "a merchant", commercial figures with "[amount]", private channel names with "[private channel]", PII with nothing.

Append qualifying vault candidates to the candidates list.

### Step 3: QMD Enrichment (only if MODE includes QMD)

Run `qmd_query` for each competency. Collections: `log`, `memory`, `learn`, `work`, `career`.

| Competency | Query |
|------------|-------|
| UX Design | `"crafted designed prototype user flow iteration hallway testing edge case coverage"` |

| Ownership | `"shipped delivered owned end-to-end product area backlog planning"` |
| Leadership | `"mentorship feedback buddying team onboarding AI literacy tooling"` |

For each QMD result:
- Check dedup (skip if already in index or already found in Step 2)
- Check date range (skip if outside {{DATE_AFTER}} to {{DATE_BEFORE}})
- Apply redaction
- Append qualifying results to candidates list

---

### Step 4: DevRev Mining (only if MODE includes DevRev)

Call `mcp__plugin_compass_devrev__hybrid_search` with queries for each competency area. Then filter by **DevRev Ownership Gate** — require at least one of:

1. the user is the **assignee** on the issue/enhancement
2. A comment authored by the user is **> 100 characters** (substantial contribution, not a reaction)
3. A **Figma or artifact URL** is linked in the issue
4. the user is listed as **owner** or **creator**

**Exclude**: passive mentions, @mention in someone else's comment, tagged for review without a substantive authored response.

For qualifying DevRev items:
- Dedup: check DevRev work ID against case.md entries AND check artifact URLs (if linked) against the primary dedup index AND check against in-run candidates from Steps 2 and 3
- Apply redaction
- Map to competency based on issue tags, linked artifacts, and content
- Append to candidates list

---

### Step 5: Slack Corroboration (only if PRIVACY_SCOPE = "all" AND MODE includes Slack)

**Window**: `max({{DATE_BEFORE}} minus 42 days, {{DATE_AFTER}})` to `{{DATE_BEFORE}}` (6-week cap — Slack `from:` is unreliable beyond 6 weeks).

Call `mcp__plugin_compass_slack-mcp__slack_search_messages` with `from:{{SLACK_DISPLAY_NAME}} after:{window_start} before:{window_end}`.

For each search result:
1. Apply **Design Counterfactual Gate** — must satisfy BOTH buckets:
   - **Bucket A — Agency evidence** (at least one of):
     - the user's input changed a decision, redirected work, or established direction — confirmed by a follow-up message or outcome, not just asserted
     - the user initiated the discussion, defined the design problem, or owned the outcome from start to finish
   - **Bucket B — Artifact evidence** (at least one of):
     - A named Figma file, design spec, user flow, prototype, or design system contribution exists that the user authored or directly produced
     - A DevRev issue or PR is linked where the user is assignee or creator

   If only one bucket is satisfied, apply Devil's Advocate and default to EXCLUDE.

2. **Role types** (Design, not engineering):
   - **Design Driver**: the user initiated, defined the problem, proposed the solution, owned the outcome
   - **Design Contributor**: Someone else initiated; the user contributed substantive design input that measurably shaped the outcome
   - **Design Observer**: the user was CC'd, reacted, or acknowledged — EXCLUDE. "Extraordinary" means Bucket A AND Bucket B are both satisfied with corroborating evidence, not just presence in the thread.

3. For threads where Bucket A or B might be satisfied: call `mcp__plugin_compass_slack-mcp__slack_get_thread_replies` to read the full thread from the beginning.

4. **Devil's Advocate (MANDATORY)**: State the strongest argument for exclusion. **Default: when in doubt, exclude.** If the exclusion argument cannot be definitively rebutted with specific evidence (artifact URL, stakeholder quote, confirmed outcome), exclude the entry. "It might have been impactful" is not a rebuttal.

5. Dedup against case.md index and existing candidates. If same achievement already found in Step 2/3/4, merge source refs — do NOT create a duplicate candidate.

6. Apply redaction. Append qualifying candidates.

---

### Step 6: Cross-Source Merge and Finalization

After all passes:
1. Scan candidates for matching dedup keys across sources
2. Merge: two candidates with same key → keep one, combine `sources[]`
3. Sort candidates by competency (UX Design first — highest weight), then by evidence strength (artifact URL present > no artifact)

### Entry Quality Standards

Every candidate entry MUST:
- Use PD-appropriate verbs: Crafted, Drove, Delivered, Championed, Validated, Advocated, Owned, Mentored, Designed, Prototyped
- NOT use: "Helped with", "Worked on", "Collaborated", "Participated in" without specifying the actual design contribution
- Name a specific artifact (Figma file, Blade Score, user flow, prototype, design system contribution, DevRev issue)
- Include at least one stakeholder handle or team reference where applicable
- Be a single natural sentence (not a fragment, not multiple sentences)
- Be framed at PD II level — demonstrates readiness for next level, not just execution at current level

Zero-candidate output is acceptable and correct when no qualifying evidence is found.

## Critical Rules

- **NEVER ask questions** — all decisions are autonomous. Use the criteria above to decide.
- **NEVER include** entries where the user was a passive observer or reviewer without substantive input
- **NEVER use** engineering vocabulary in entry text
- **NEVER skip** the Devil's Advocate step for Slack entries — "when in doubt, exclude" is the default
- **NEVER write directly to** `career/case.md` — output goes to `case-candidates.md` only
- **NEVER include** raw Slack thread body text — one-line entry + source ref only
- **Zero candidates is correct** when evidence doesn't meet the bar — do not force-include weak entries
- **Apply redaction** before writing any candidate text
- **Re-check all finalized candidates** against Critical Rules before writing Step 8 output
